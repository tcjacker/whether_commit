# Review Graph 测试关联规范（草案）

## 1. 目的

本规范用于支撑 `review graph` 中的测试与代码对象关联。

本规范优先解决的问题不是统一工程目录，也不是统一文件命名，而是先回答下面两个问题：

1. 一个测试主要在证明哪个主对象。
2. 当代码发生变更时，系统应如何稳定地把测试证据挂到对应的 `Feature` 或 `Code` 节点上。

因此，本规范的第一优先级是约定 **测试与主对象的关系**。目录结构、文件命名、语言差异和历史遗留代码，只作为增强信号处理，不作为前提条件。

## 2. 适用范围

本规范适用于 `review graph` 需要消费的测试证据，包括但不限于：

- HTTP API 的测试
- job / scheduler / consumer 的测试
- service / domain logic 的测试
- 集成测试
- 回归测试

本规范默认兼容：

- 不同编程语言
- 不同测试框架
- 新代码与存量代码并存
- 无法大规模调整目录结构和文件命名的项目

## 3. 核心原则

1. 先约定测试与主对象关系，再考虑工程组织信号。
2. 一个测试必须能回答“它主要在证明谁”。
3. 主对象关系应尽量显式，不能完全依赖猜测。
4. AST、导入、调用链、目录、文件名都只应作为辅助证据。
5. `review graph` 默认优先信任显式关系，其次才是静态分析和命名推断。

## 4. 对象定义

### 4.1 主对象

主对象是一个测试主要要证明的代码对象，也是 `review graph` 默认应挂接的目标。

主对象可以是：

- 入口宿主容器
  - 例如 API class、router、controller、server、job handler、consumer handler、scheduler handler
- 核心代码对象
  - 例如 class、function、method、module

示例：

- `OrdersApi`
- `PaymentRetryConsumer`
- `OrderService.create_order`
- `retry_policy` 模块

### 4.2 补充对象

补充对象是测试在验证主对象时顺带覆盖到的相关代码对象。

补充对象可以用于丰富证据，但默认不作为主挂接目标。

### 4.3 推断对象

推断对象是系统通过 AST、导入关系、调用关系、目录结构、文件命名等手段推导出的弱关联对象。

推断对象只能作为低置信度证据，不应直接覆盖显式主对象关系。

## 5. 关联强度

建议将测试与代码对象之间的关系分成三类：

- `primary`
  - 主测试对象
  - 进入 `review graph` 的主证据层
- `secondary`
  - 补充覆盖对象
  - 可在详情面板中作为补充证据展示
- `inferred`
  - 通过静态分析或命名规则得到的弱关联
  - 默认不进入主图，只在需要时展开

默认消费顺序如下：

1. `primary`
2. `secondary`
3. `inferred`

## 6. 最小约束：每个测试必须声明或可稳定归约到主对象

这是本规范的核心要求。

每个测试必须满足下列条件之一：

1. 显式声明主对象
2. 能通过稳定规则无歧义归约到主对象

如果一个测试既没有显式声明主对象，也无法通过稳定规则归约，则该测试只能作为低置信度证据，不应直接进入 `review graph` 主层。

## 7. 推荐的显式声明机制

为了兼容不同语言、不同框架和存量代码，推荐支持轻量显式声明，而不是强制统一目录结构。

### 7.1 文件级声明

示例：

```python
# review-graph: primary=backend.app.api.orders.OrdersApi
# review-graph: secondary=backend.app.services.order_service.create_order
```

### 7.2 类级声明

示例：

```python
class TestOrdersApi:
    """review-graph: primary=backend.app.api.orders.OrdersApi"""
```

### 7.3 用例级声明

示例：

```python
def test_orders_create_no_options():
    # review-graph: primary=backend.app.api.orders.OrdersApi
    ...
```

### 7.4 通用要求

显式声明机制只需要表达三件事：

- `primary`
- 可选的 `secondary`
- 被引用对象的稳定标识

对象标识不要求必须是某一种语言的全限定名，但必须是仓库内稳定、可解析、可追踪的标识。

## 8. 无显式声明时的稳定归约规则

如果项目暂时不引入显式声明，系统可以按以下优先级推断主对象，但必须保证可解释。

推荐优先级：

1. 测试类或测试套件的主对象名
2. 测试函数名中的主对象名
3. 测试文件中最主要被实例化、调用或 patch/mock 的对象
4. 测试文件导入中最强命中的对象
5. 文件命名约定
6. 目录镜像关系

注意：

- 目录结构和文件命名在这里是第 5、6 级信号，而不是第 1 级。
- 如果多个对象同时命中，且无法稳定区分主对象，则结果应标记为 `inferred` 而不是伪装成强关联。

## 9. 对命名与目录的定位

命名与目录不是本规范的前提条件，只是增强信号。

也就是说：

- 不要求项目必须重构目录结构
- 不要求存量代码必须改文件名
- 不要求所有语言都遵守同一套目录镜像约定

但如果项目愿意配合，以下信号会提升 `review graph` 质量：

- 测试类名显式对应主对象
- 测试函数名显式包含对象名与行为名
- 测试文件名与主对象接近
- 测试目录与源码目录大致镜像

这些都属于推荐项，而不是硬约束。

## 10. 推荐命名信号

虽然不作为硬前提，但命名仍然是低成本高收益的增强项。

### 10.1 测试类

推荐：

- `TestOrdersApi`
- `TestPaymentRetryConsumer`
- `TestOrderService`

### 10.2 测试函数

推荐格式：

```text
test_<object>_<behavior>
```

例如：

- `test_orders_create_success`
- `test_orders_create_no_options`
- `test_payment_retry_skips_duplicate_event`

不推荐：

- `test_success_case`
- `test_basic_flow`
- `test_edge_case_1`

原因不是风格问题，而是这些名称对 `review graph` 的对象归属没有帮助。

## 11. 不同类型测试的主对象规则

### 11.1 入口测试

针对 API、job、scheduler、consumer 的测试，主对象应优先是入口宿主容器。

例如：

- `OrdersApi`
- `PaymentRetryConsumer`
- `OrderSyncJob`

### 11.2 service 测试

针对 service 或 domain logic 的测试，主对象应优先是：

- class
- method
- function
- module

选择哪一级，取决于项目当前是否有稳定对象边界。

### 11.3 集成测试

集成测试可以覆盖多个对象，但仍应有一个主对象。

如果没有单一主对象，则不应强行挂到某一个 `Feature`，而应：

- 标记为跨对象测试
- 仅作为 `secondary` 或 `inferred` 证据参与展示

### 11.4 回归测试

回归测试应优先挂到引发回归风险的主对象，而不是挂到所有参与对象。

如果回归测试确实跨多个对象，建议使用显式 `primary/secondary` 标注。

## 12. AST 与静态分析的角色

AST 和静态分析是补强工具，不是主规范。

它们适合做以下事情：

1. 识别测试导入了哪些对象
2. 识别测试调用了哪些函数或方法
3. 识别测试 patch/mock 了哪些对象
4. 识别测试中显式引用了哪些 route、job、consumer 名称
5. 在无显式声明时，帮助生成 `inferred` 关系

它们不适合做以下事情：

1. 覆盖显式的 `primary`
2. 在多对象混合测试中武断认定唯一主对象
3. 用复杂推断掩盖工程组织混乱

## 13. Review Graph 的默认消费规则

后端聚合器建议按如下顺序消费测试关系：

1. 读取显式 `primary`
2. 读取显式 `secondary`
3. 若无显式声明，则按稳定归约规则推导主对象
4. 若仍不明确，则使用 AST / 静态分析生成 `inferred`
5. 若关联歧义过大，则不进入主图，只作为补充证据

对应展示策略：

- `primary`
  - 进入默认主图或主详情证据区
- `secondary`
  - 进入详情面板补充证据区
- `inferred`
  - 默认折叠，仅在需要时展开

## 14. 最低落地要求

如果要快速落地第一版，建议只落实以下要求：

1. 每个测试必须存在主对象
2. 主对象优先通过显式声明表达
3. 没有显式声明时，只做保守归约
4. 目录和命名只作为辅助信号，不作为前提约束
5. 无法稳定判定的测试，不直接进入主图

这是第一版最重要的边界。它比统一工程结构更现实，也更兼容存量项目。

## 15. 反模式

以下模式会显著伤害 `review graph` 的可信度：

1. 一个测试覆盖多个对象，但没有主对象声明
2. 系统在弱证据下强行认定主对象
3. 用目录结构猜测去覆盖显式声明
4. 将所有集成测试都当成单对象强证据
5. 过度依赖 AST 推断来替代主对象约定

## 16. 团队建议

团队如果要推动这件事，建议优先做下面三件事，而不是先改目录：

1. 约定每个测试必须有主对象
2. 约定复杂测试支持 `primary/secondary` 显式声明
3. 约定后端聚合器始终优先消费显式关系

等这三件事稳定之后，再决定是否逐步引入更强的命名约定、文件组织约定或 AST 自动分析。

## 17. 后续演进顺序

推荐按以下顺序演进：

1. 先建立主对象约定
2. 再支持显式 `primary/secondary` 标注
3. 再实现保守的静态归约
4. 再增加 AST import/call/mock 补强
5. 最后再考虑目录和命名规范的优化建议

顺序不能反过来。否则系统会先被工程结构差异和历史包袱绑住，而不是先获得稳定、可解释的测试归属模型。
